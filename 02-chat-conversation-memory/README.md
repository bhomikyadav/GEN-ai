# 💬 Chapter 02 - Chat Conversation Memory

> Learn how to build a conversational AI application by maintaining chat history (context) using the OpenAI Chat Completions API.

---

# 📖 Overview

In the previous chapter, we sent a single prompt to the model and received a single response. Every API request was independent, meaning the model had **no memory** of previous interactions.

In this chapter, we'll build a simple terminal-based chatbot that maintains the conversation history. By sending the entire conversation with every request, the AI can understand context and respond naturally, just like a real chat application.

This chapter introduces one of the most important concepts in Generative AI: **Conversation Context**.

# ❓ Why Do We Need Conversation Memory?

Imagine the following conversation:

```text
User: My name is Bhomik.

Assistant: Nice to meet you, Bhomik!

User: What's my name?
```

If we only send the second message:

```text
What's my name?
```

the model has no idea who "my" refers to because it has no knowledge of previous messages.

Instead, we send the complete conversation:

```text
System:
You are a helpful assistant.

User:
My name is Bhomik.

Assistant:
Nice to meet you, Bhomik!

User:
What's my name?
```

Now the model has enough context to answer:

```text
Your name is Bhomik.
```

This is exactly how most AI chat applications work.

# 🧠 Core Concepts

This project introduces several important concepts.

## 1. OpenAI Client

The OpenAI client is responsible for communicating with the OpenAI API.

```ts
const openai = new OpenAI({
    apiKey: process.env.OPEN_AI_API_KEY,
});
```

---

## 2. Context Array

The conversation is stored inside an array.

```ts
const context = [];
```

Each object represents a message.

```ts
{
    role: "user",
    content: "Hello"
}
```

The context grows as the conversation continues.

---

## 3. Message Roles

Every message belongs to one of three roles.

### System

Provides instructions to the model.

Example:

```text
You are a helpful assistant.
```

The system message controls the AI's behavior throughout the conversation.

---

### User

Represents messages sent by the user.

Example:

```text
What is TypeScript?
```

---

### Assistant

Represents previous responses generated by the AI.

These responses are stored so the model remembers what it previously said.

# ❌ Limitations

This approach stores **every message** in memory.

As the conversation grows:

- More tokens are sent with every request.
- API costs increase.
- Responses become slower.
- Eventually the model reaches its context window limit.

Because of these limitations, production applications usually implement:

- Conversation trimming
- Conversation summarization
- Long-term memory
- Vector databases
- Retrieval-Augmented Generation (RAG)

These topics will be covered in later chapters.

---

# 🏭 Production Considerations

This implementation is designed for learning purposes.

For production applications, consider:

- Error handling using `try...catch`.
- Validating API responses.
- Limiting conversation size.
- Storing conversations in a database.
- Streaming responses instead of waiting for completion.
- Logging requests and responses.
- Rate limiting.
- Protecting API keys.
- User authentication.

---

# 💡 Best Practices

- Keep the system prompt concise.
- Never expose API keys.
- Store chat history securely.
- Remove unnecessary messages from the context.
- Validate user input.
- Handle API failures gracefully.
- Use descriptive variable names.
- Separate AI logic into reusable services.

---

# ⚠️ Common Mistakes

❌ Forgetting to append the assistant response.

❌ Sending only the latest user message.

❌ Hardcoding API keys.

❌ Ignoring API errors.

❌ Allowing the context array to grow indefinitely.

❌ Forgetting the system message.
