# 📚 What is Function Calling?

Large Language Models (LLMs) cannot execute JavaScript functions, access databases, call APIs, or retrieve real-time information on their own.

Instead, they can **request** that your application execute a function on their behalf.

Think of it like this:

```
User
 │
 ▼
LLM
 │
 ▼
"I need external information."
 │
 ▼
Your Application
 │
 ▼
Execute Function
 │
 ▼
Return Result
 │
 ▼
LLM
 │
 ▼
Final Answer
```

The model never executes your code. It only decides **when** a function should be called.

# 1️⃣ Importing Required Packages

```ts
import { OpenAI } from "openai";
import { configDotenv } from "dotenv";
```

## Purpose

Import the required libraries.

### OpenAI

Provides the official SDK for interacting with OpenAI models.

### dotenv

Loads environment variables from the `.env` file into `process.env`.

---

# 2️⃣ Loading Environment Variables

```ts
configDotenv();
```

## Why?

Reads the `.env` file and makes variables available throughout the application.

Example:

```env
OPEN_AI_API_KEY=sk-xxxxxxxx
```

After loading:

```ts
process.env.OPEN_AI_API_KEY;
```

contains the API key.

Without calling `configDotenv()`, the API key would be `undefined`, and every request would fail.

---

# 3️⃣ Creating the OpenAI Client

```ts
const openai = new OpenAI({
  apiKey: process.env.OPEN_AI_API_KEY,
});
```

## Purpose

Creates an authenticated client that communicates with the OpenAI API.

Every request made by the application passes through this client.

---

# 4️⃣ Creating the Local Function

```ts
function getTimeOfIndia() {
  return new Date().toLocaleString("en-IN", {
    timeZone: "Asia/Kolkata",
  });
}
```

## Purpose

This function returns the current date and time in the Indian Standard Time (IST) timezone.

The model **cannot call this function directly**.

Instead, it requests that your application execute it.

---

# 5️⃣ Creating the Conversation Context

```ts
let context = [
  {
    role: "system",
    content: "...",
  },
  {
    role: "user",
    content: "What is current time",
  },
];
```

## Purpose

The context array stores the complete conversation history.

Every API request sends this array to the model.

Initially it contains:

```
System

↓

User
```

As the conversation grows, additional messages are appended.

---

# 6️⃣ Understanding the System Message

```ts
role: "system";
```

The system message defines the assistant's behavior.

Example:

```
You are a helpful assistant.
```

It influences every response generated by the model.

---

# 7️⃣ Starting the AI Request

```ts
const res = await openai.chat.completions.create(...)
```

This sends the first request to the model.

At this stage, the model decides one of two things:

```
Can I answer myself?

OR

Do I need a tool?
```

---

# 8️⃣ Registering Tools

```ts
tools: [
    {
        type: "function",
        function: {
            ...
        }
    }
]
```

## Purpose

Registers functions that the model is allowed to request.

These are not executed automatically.

Instead, the model becomes aware that they exist.

---

# 9️⃣ Understanding Tool Metadata

Every function contains:

```ts
name;
```

The unique function identifier.

---

```ts
description;
```

Explains when the function should be used.

This is one of the most important fields.

The model reads this description before deciding whether to invoke the tool.

---

```ts
parameters;
```

Defines the function's expected input.

Your function has no parameters.

Therefore:

```ts
properties: {
}
```

is sufficient.

---

# 🔟 tool_choice

```ts
tool_choice: "auto";
```

## Purpose

Allows the model to decide whether a tool is necessary.

Possible values:

```
auto
```

Model decides.

---

```
none
```

Never call a tool.

---

Specific function

Always call the specified function.

---

# 1️⃣1️⃣ First Response

```ts
const res = ...
```

Possible outcomes:

### Normal Response

```
Hello!
```

---

### Tool Call

```
tool_calls
```

If the model returns `tool_calls`, it is asking your application to execute a function.

---

# 1️⃣2️⃣ Checking Whether a Tool Was Requested

```ts
finish_reason === "tool_calls";
```

The API indicates that the response was intentionally stopped because the model wants a function to be executed.

This is your signal to begin tool execution.

---

# 1️⃣3️⃣ Reading Tool Calls

```ts
const toolCall = res.choices[0]?.message.tool_calls?.[0];
```

The assistant message may contain one or more tool requests.

Example:

```json
{
  "tool_calls": [
    {
      "id": "...",
      "type": "function",
      "function": {
        "name": "getTimeOfIndia"
      }
    }
  ]
}
```

---

# 1️⃣4️⃣ Executing the Function

```ts
if (toolCall.function.name === "getTimeOfIndia") {
  const time = getTimeOfIndia();
}
```

Now your application executes the JavaScript function.

Remember:

The model requested it.

Your application performs the actual execution.

---

# 1️⃣5️⃣ Saving the Assistant Message

```ts
context.push(res.choices[0].message);
```

This assistant message contains the tool request.

It must be included in the conversation before adding the tool result.

Without it, the conversation becomes invalid because the model cannot associate the tool response with its original request.

---

# 1️⃣6️⃣ Returning the Tool Result

```ts
context.push({
  role: "tool",
  content: time,
  tool_call_id: toolCall.id,
});
```

This message tells the model:

```
The requested function has been executed.

Here is the result.
```

The `tool_call_id` links the tool output to the corresponding tool request.

---

# 1️⃣7️⃣ Second API Call

```ts
const secondCall =
await openai.chat.completions.create(...)
```

## Why is a second request required?

The first request only determines whether a tool is needed.

The second request provides the tool result back to the model.

Only then can the model generate the final answer.

---

# 1️⃣8️⃣ Final Response

Example:

```
The current time in India is 8:45 PM.
```

The model combines:

- User question
- Tool output
- Conversation history

to generate a natural response.

# ✅ Advantages

- Access real-time information.
- Execute custom business logic.
- Connect to databases.
- Call REST APIs.
- Retrieve weather, stock prices, or user data.
- Build intelligent assistants with external capabilities.

---

# ❌ Limitations

- The model cannot execute JavaScript directly.
- Every tool must be implemented by the developer.
- Requires at least two API requests when a tool is invoked.
- Tool descriptions must be accurate for reliable invocation.

---

# 🏭 Production Improvements

Before using this pattern in production, consider:

- Wrap API calls in `try...catch`.
- Validate the API key during startup.
- Validate tool arguments before execution.
- Add authentication for sensitive tools.
- Log tool usage and failures.
- Handle unknown tool names gracefully.
- Use a centralized Tool Registry instead of multiple `if` statements.
- Return structured JSON instead of plain text where appropriate.
- Monitor execution time and failures.

---

# 💡 Best Practices

- Use descriptive function names.
- Write clear and specific tool descriptions.
- Keep each function focused on a single responsibility.
- Validate all incoming parameters.
- Separate tool logic from AI orchestration.
- Design tools to be reusable and testable.

---

# ⚠️ Common Mistakes

❌ Assuming the model executes functions automatically.

❌ Forgetting the second API request.

❌ Not appending the assistant tool-call message before the tool response.

❌ Ignoring `tool_call_id`.

❌ Using vague or misleading tool descriptions.

❌ Executing unvalidated tool arguments.
